Overview

You have a need to verify that your customers are who they say they are. We know our customers well; we have to! It’s the law. So let us help: use the Sign Up with Capital One API.

Here’s how it works: your customers begin by registering for your site using their Capital One credentials and other security procedures; once their credentials are verified and they’ve given consent, we’ll share with you the identifying information (name, address, date of birth, and so on) they’ve permitted us to. After users are registered, they can continue to log in to your site using their Capital One credentials (using Sign In with Capital One), or you can create a profile and a new set of credentials specific to your site. Either way, you can be confident you know who you’re dealing with.

And this knowledge can become part of the service you offer: for any arrangement that requires two of your users to connect, you can assure each user that the other has undergone a rigorous identity verification process, but neither has to share sensitive personal data with a total stranger. Win-win-win!

Protecting personal data has never been more critical; assure your customers that you’re doing it right by partnering with Capital One.

Hello world

The Sign Up with Capital One API allows you to access our implementation of the OAuth 2.0 standard for authorization, and use it to verify the identity of a user who is registering for your site. For this product you’ll make three calls:

  • first (using a browser), to our /oauth2/authorize endpoint, which prompts the end user to log in and provide consent;

  • second, to our /oauth2/token endpoint (using the authorization code returned by the /authorize call) to retrieve an access token; and

  • third, to our /oauth2/userinfo endpoint to exchange the access token returned by the /token call for any personal data the end user has agreed to share with you.

In a production environment, you’ll likely want to take advantage of the Retrieve sign up button code operation; this operation returns the HTML code you need — fully customized and ready to go — to add a Sign Up with Capital One button to your experience. A click on that button will initiate the /oauth2/authorize call for you, and prompt your end-user to log in. After that, you just grab that authorization code and carry on!

For our simple cURL-based Hello World, however, a button isn’t helpful, so here we’ll walk you through the specific calls. But before you can begin your Capital One product manager must provide you with access to a co-dev application and make sure it is properly configured; all you have to do is to sign up for a Capital One DevExchange account. You can find more information about how to do this in Getting Started.

Note

For simplicity’s sake, we’ll ignore certificate errors by passing the '-k' argument in these test commands. You would of course properly handle your certificates in production.

Ready to move on? Let’s go.

  1. Gather your credentials

    You’ll need the following information:

    • Your Client ID, Client Secret, and Redirect URI, which are assigned to your co-dev app. You can find them on the Co-Dev tab of your App Details page.

      Note
      		 You can also use the sandbox server (https://api-sandbox.capitalone.com) and your app’s Sandbox credentials to make these calls; just be sure to use a test account appropriate to the server you’ve chosen. The cURL provided below shows the call to the co-dev server; be sure to adjust the code if you choose to use the sandbox server instead.

    • The appropriate scope(s) for the call to /oauth2/authorize. Two scopes are required:

      • signup

      • openid

    • (Optional but strongly recommended) The value of a state parameter. You pass this value to the authorization server in your initial authorization request, and the server returns it when it redirects back. This value is important for protecting against cross-site request forgery (CSRF) attacks.

      You should construct the state value as a unique non-guessable string. See the OAuth 2.0 specification for more information.

  2. Initiate the OAuth process.

    First, you call our authorization server, which will prompt you (or, later, your customer) to log in to a Capital One account. You’ll need your Client ID and Redirect URI — and the required scopes — to construct the appropriate URL, which you can use in any browser:

    https://apiit.capitalone.com/oauth2/authorize?client_id=YOUR_CLIENT_ID&redirect_uri=YOUR_REDIRECT_URI&scope=signup openid&response_type=code&state=YOUR_STATE_VALUE

    This launches a Capital One login screen; you can log in using one of the test accounts provided. Note that, in a production environment, Sign Up may require two-factor authentication; we’ve provided two test accounts that require this. After you log in with one of these accounts, you must perform one additional step: either

    • request a one-time password (OTP), which you must enter in the screen that displays; or

    • respond to a SwiftID notification on a mobile device.

    Note
    		 You’ll need some help from us to get set up to respond to the two-factor authentication; please coordinate with your Capital One product manager for assistance.

    Once login is complete, you must approve the sign up request:

    If consent is provided, an authorization code is appended to your Redirect URI, together with the state parameter value if you included one, like this:

    https://YOUR_REDIRECT_URI?code=YOUR_AUTHORIZATION_CODE&state=YOUR_STATE_VALUE
    Important
    		 If you provide the state value, be sure to check the returned value against the original value that you sent before proceeding to the next step. If the values do not match, the request is a forgery, and you should go no further. Ignore the request, and file a support ticket.

Servers

The Capital One DevExchange platform provides three servers that serve distinct purposes. You might start off using the sandbox server, which provides a fixed set of use cases and accompanying test data:

https://api-sandbox.capitalone.com

then move on to the co-dev test server, which provides a broader set of test data and allows for testing a more fluid set of scenarios:

https://apiit.capitalone.com

When you’re ready to go live, you’ll use the production server:

https://api.capitalone.com

You can find the credentials you need for calling the codev and sandbox servers on your App Details page. You’ll need special credentials in order to access the production server; contact your Capital One product manager for details.

Versioning

When you make a Capital One API call, you must declare the version of the endpoint that you want to invoke. The version is declared by appending a v=N element to the Accept header, where N gives the major version number. For example:

Accept: application/json; v=1

Version numbers are always whole numbers, and are always specified by major version only; if a minor version of the API exists, your call will automatically go to the highest-available minor version. If you call a version that does not exist, you’ll get a 403 status code in response.

Obtaining an access token

In the client credentials (or two-legged OAuth) scheme, you trade your Client ID and Client Secret directly for the access token in a request that looks like this:

Managing bad tokens

An access token can become invalid for various reasons — it may expire, or something can go wrong on the authorization server.

If this happens, you’ll have to request a new token using your Client ID and Client Secret as you did at the beginning.

Gateway errors

Our API gateway defines the following errors that are shared across products. Errors that are unique to individual products are documented in the Response: Error Codes section for each operation.

When a Capital One API returns a 4xx or 5xx HTTP status, the response body contains an Error Response Structure that describes the error that caused the fault:

  • code (integer) — A six-digit integer that identifies the cause of the error. You can feed the code value into an error handler or use it to otherwise predicate your code.

  • description (string) — A plaintext, human-readable description of the error. The audience for this string is you, the app developer; you mustn’t display the message to the end-user.

  • documentationUrl (string) — The URL of the developer documentation that directs you to more information about the error. This URL is provided as an aid that you can use during development; you mustn’t display the URL to the end-user.

Only one error is returned per API invocation.

Note

Capital One OAuth endpoints

The Sign Up with Capital One API product provides access to three OAuth endpoints implemented by Capital One:

  • Retrieve authorization code — prompts the end user to log in using Capital One credentials; once login is successful and the user has consented to the use of their data, returns an authorization code.

    You can call this endpoint directly, or use the Retrieve sign up button code operation to obtain fully-customized and ready-to-use HTML code you can insert in your experience to provide a Sign Up with Capital One button; when this button is clicked the Retrieve authorization code call is made for you.

  • Create access token — allows the caller to exchange the authorization code for a set of tokens, one of which — the access token — you’ll need to provide in your call to Retrieve user information.

  • Retrieve user information — returns personally identifying information (name, address, phone, etc) about the user who provided consent in the Retrieve authorization code call.

Once you’ve made all three calls, you’ve verified that your new user is a Capital One customer, and you’ve obtained important information (with the user’s consent) that allows you to identify the specific human behind the account.

You can use these endpoints to allow Capital One customers to register for your app or website using their Capital One credentials, and use their identifying data to create a new account on your site or perform other identity verification as needed.

Retrieve authorization code

This operation is used to begin the three-legged OAuth flow — the one that requires end-user log in and consent. Typically, the goal of the three-legged OAuth process is to obtain an access token that can be used to authorize calls to another API; this is also true in the Sign Up with Capital One workflow, though the API for which the access token is used is also an OAuth endpoint.

Retrieve authorization code initiates the Capital One login sequence, and prompts users to consent to sharing their data; once the log in has succeeded and consent has been granted, the operation returns an authorization code that can be exchanged for an access token.

Note that you can use the Sign Up with Capital One button — the code for which is returned by the Retrieve sign up button code operation — to make this call for you.

GET /oauth2/authorize

Request

Query Parameters
client_id* string

The Client ID that is assigned to your co-dev app. You can find this value on your App Details page.

redirect_uri* string

The URI to which your authorization code will be returned. This value must match the Redirect URI value assigned to your co-dev app (see your App Details page).

scope string

The authorization scope granted by the resource owner. The call to this operation must include two space-delimited scope values:

  • signup

  • openid

response_type* string

Declares the type of information that you want to have sent back in the response. This value must be set to code, which indicates that an authorization code is the expected response.

state string

An optional but recommended value you can provide as part of your request. The actual value is arbitrary and can be anything you want; it’s simply returned, with the authorization code, to the defined Redirect URI.

You can use this value like a checksum: comparing the state value that you include in the request with the value that’s passed back in the response can help protect you against cross-site request forgery (CSRF) attacks.

HTTP Status Codes
